# PRD — stack-adaptive bug loop MVP (`detect-stack` + generic `investigate-bug`)

**Status:** ❄️ FROZEN 2026-06-07 (user-approved; `prd-review` PASS — 6 blockers fixed incl. the parent/child inversion + P4 write-phase + menu-count, all verified at source) · **Slug:** `stack-adapters`
**Freeze:** Scope held at PR-A + PR-B; changes require an explicit amendment. `apex:design-feature` may begin (resolves U2 adapter-contract shape via `apex:api-surface-review`).
**Scope:** MVP = **PR-A + PR-B only**. Source: `docs/stack-adapters/brief.md` (decisions baked in 2026-06-07 — not re-litigated here). Format + depth exemplar: `docs/incident-retro/prd.md`.
**Composes (does not duplicate):** `autonomous-fix` (the shipped write-safety gate; SKILL shipped, PRD frozen). This effort is the **read / diagnose / route** layer that feeds that gate's write phase; it re-implements **none** of its rails.

---

## 1. Problem

apex's bug-fix story has a shipped **write-safety gate** (`autonomous-fix`: read-only-first, reproduce-first, sensitive-path refuse, fail-closed cost cap, draft-PR-only) but **no tool-neutral layer in front of it that knows how to *reach* the bug across a heterogeneous stack**. Adaptive bug-triage products exist and are widely deployed (Sentry Seer, Linear/Jira agents, GitHub Copilot coding agent — see §9), but **each is welded to its own tracker + its own observability + its own single fix backend**. None route across *whatever combination* a project already runs. The combinatorics are the whole problem: trackers (GitHub/Linear/Jira/…), observability backends (Datadog/Sentry/OTEL/console), and reproduce surfaces (compose/staging/CI) multiply to ~288 stack combinations. Enumerating integrations doesn't scale; apex must **abstract over the axes and route through whatever a project already has**, shipping zero integrations of its own — and hand the write to an **auditable, in-repo** gate, not an opaque vendor one.

The narrow, defensible gap: not "nothing reaches the bug" but **"nothing reaches the bug tool-neutrally, across a heterogeneous stack, handing to an auditable in-repo write gate."**

A second, sharper gap: the one BookBridge `investigate-bug` skill that exists is **project-specific** (its F-049 DB queries, its hypothesis matrix). The investigation discipline — fetch the issue, query the failure window, find suspect commits, reproduce, hand to the gate — is **generic**, but apex has never owned the generic form. So every consumer re-derives it.

This PRD specifies the smallest layer that closes both gaps: **`detect-stack`** (a config step that probes a project and writes a routing-only `apex.profile.toml`) and a **generic `apex:investigate-bug`** (reads the profile, dispatches to the configured adapter, runs the gate's read-only investigate + reproduce phases as the gate's own rails, then **hands the reproduced bug into `autonomous-fix`'s write phase**). The MVP ships the **two universally-available adapters** — GitHub-tracker via the `gh` CLI, console-logs via `grep` — to prove the adapter shape with backends every repo has, before any vendor adapter is built.

This is **abstraction-over-enumeration** — and over *agents*: apex defines the capability contract; the user plugs in what they have; apex routes (and could one day route the *write* to a vendor agent too — §9).

## 2. Goal & acceptance criteria

A **two-skill MVP**: (a) **`apex:detect-stack`** + `/apex:detect-stack` — probes deps / env names / CI configs / connected MCP servers, writes an in-repo **routing-only** `apex.profile.toml`; (b) **generic `apex:investigate-bug`** — reads that profile, dispatches per axis to the configured adapter, runs the gate's read-only investigate + reproduce phases, and **composes** `autonomous-fix` for the gated write. The MVP ships exactly **two adapters** (GitHub-tracker / `gh`, console-logs / `grep`). It re-implements **none** of `autonomous-fix`'s write rails.

**The two seams this PRD must nail (stated up front so design doesn't blur them):**

- **The compose-with-`autonomous-fix` contract (AC6).** `investigate-bug` is the adapter-routed diagnosis that **instantiates the gate's read-only phases and then feeds the gate's write phase** — exactly the `investigate-bug → autonomous-fix` edge the autonomous-fix PRD already names. The handoff point is **precise and verified against the gate's phase structure** (`autonomous-fix/SKILL.md`): `investigate-bug` runs **P2** (read + emit `PLAN.json` + write the reproducing test under the gate's **B2 test-file-only carve-out**) and **P3** (`assert-red-test` — red on the unpatched tree), as the gate's own read-only rails, then the write-side handoff is **into P4**. **The seam is P3→P4.** From P4 on, every write-side rail (sensitive-path refuse, nonce fence, fail-closed cost cap, draft-only, gate-owned commit) is the gate's, **unchanged**. Because `investigate-bug` operates *inside* the gate's read-only allowlist (write tools absent except the declared test path), "writes no source" is **inherited from the gate's staged allowlist**, not asserted by a markdown skill that cannot enforce it. `investigate-bug` MUST NOT open a PR, write source, or re-decide any write-side rail.
- **The interactive-vs-unattended adapter seam (AC2 + AC3).** Session MCP (`mcp__<vendor>__*` via ToolSearch) exists **only in an interactive session**. The unattended CI path — where `autonomous-fix` runs — has **no session MCP** and must use CLI/HTTP. So **every *routed* axis in the profile carries TWO bindings** (`mcp` for interactive, `cli` for unattended) plus a hard fallback (ask the user to paste). This is an **API-surface concern** → `apex:api-surface-review` runs at design time on the profile schema + the adapter contract, **before PR-A is cut**.

**Enforcement-surface note (mirrors `autonomous-fix/prd.md`'s table):** `detect-stack` and the routing **selector** are real code with **mechanical** enforcement (unit-testable). AC5/AC6's read-only and no-bypass properties are **advisory at the investigate-bug markdown layer; the real enforcer is the gate's two-invocation allowlist** that investigate-bug runs inside. Each AC below names its surface.

**Acceptance criteria (observable):**

- **AC1 — `detect-stack` writes a routing-only profile, NO secrets [mechanical].** `apex:detect-stack` probes the repo (deps, env-var *names*, CI configs, connected MCP servers) and writes/updates `apex.profile.toml` containing **only routing config**. Decidability is a **value-shape allowlist**: every field's value must match its declared kind's expected shape — `kind` ∈ a fixed enum, `mcp` ∈ an `mcp__*` prefix, `cli` ∈ a bare command name, `service`/`repo`/`host` ∈ an identifier, **`secret_ref` ∈ an env-var NAME `[A-Z][A-Z0-9_]*`** (never a value), `bug_label`/repro commands ∈ free strings. A profile with **any field value that fails its field's shape** (e.g. a token-shaped value where an identifier is expected, or a `secret_ref` that isn't a NAME) **fails AC1**. This makes "no credential value" *decidable*, not a regex guess. *(Boundary: the `autonomous-fix` AC8 no-leak invariant applied to the config file; the gate's AC8 detector is project-supplied — this PRD supplies the value-shape rule for the profile.)*
- **AC2 — Each *routed* axis declares BOTH a `mcp` and a `cli` binding [mechanical].** For every **routed** axis it writes (`tracker`, `observability`), the profile records an **interactive binding** (`mcp`, may be empty) **and an unattended binding** (`cli`, may be empty), and **never omits the field**. AC2 splits into two properties: **(a) field-presence** — both fields always present, empty allowed; **(b) no-silent-drop** — given a fixture environment manifest declaring what is available, every *available* binding is recorded (the detector must not drop a binding that exists). `vcs` (a descriptor field — `git`/`gh` is the universal substrate, no adapter) and `reproduce` (a command field — a command to run, not a tool to route) are **explicitly exempt** from the two-binding rule. *(This is the interactive/unattended seam, made a hard profile-schema property — scoped to the axes that actually route.)*
- **AC3 — `investigate-bug` selects the binding by mode, MCP-first interactive, CLI-only unattended [mechanical].** Interactively, it routes MCP-first, falls back to CLI, then to ask-user. On the **unattended path** (the `autonomous-fix` CI context, no session MCP), it routes **CLI/HTTP only** — it MUST NOT attempt an MCP/ToolSearch binding. A run that tries to resolve an `mcp__*` prefix in the unattended path **fails**. Hard fallback in both modes: if neither binding resolves, ask the user to paste (interactive) or escalate (unattended). *(Maps 1:1 onto `autonomous-fix`'s interactive/autonomous two-mode split.)*
- **AC4 — Two universal adapters ship, each conforming to its axis's contract [mechanical].** The MVP ships exactly two adapters: **GitHub-tracker** (`fetch_issue` / `comment` via `gh`) and **console-logs** (`query_logs(window)` via `grep` over a configured path). Each implements **that axis's capability contract** (a tracker exposes `fetch_issue` + `comment`; an observability source exposes `query_logs(window)`) — whether the contracts unify into one interface or stay per-axis is **U2** (api-surface-review). A third adapter added later MUST satisfy the same axis contract with no change to `investigate-bug`. A `kind` with no shipped adapter routes to the **ask-user hard fallback**, never to a stub that silently returns nothing.
- **AC5 — `investigate-bug` is read-only until P3→P4 handoff; it writes no source [advisory at the skill layer; enforced by the gate's read-only allowlist].** Running the gate's P2+P3, `investigate-bug` reads the tracker, reads logs, reads suspect commits, and produces a reproducing test — but writes **no source code** and opens **no PR**. The single write is the **reproducing test file**, and it is **the gate's B2 carve-out exercised inside the gate's invocation #1** (write tools absent except the declared test path) — not an independent investigate-bug privilege. A flow where investigate-bug edits source or opens a PR **fails**. *(Generalizes BookBridge `investigate-bug` **Rule #4 (no fix code)** — the no-source-write boundary; identical to gate P2/AC7b. BookBridge Rule #1 is SELECT-only DB, a separate DB rule the gate already generalized at `SKILL.md:62`, not re-claimed here.)*
- **AC6 — The handoff feeds the gate; it does not bypass or duplicate it [advisory at the skill layer; enforced by the gate].** At the P3→P4 handoff, `investigate-bug` passes the reproducing red test **+ `PLAN.json` (the planned source-file set, emitted by the runner in invocation #1 so the gate's Stage-A sensitive check has its input) + a fenced bug-context bundle** (issue body/logs as data-not-instructions, per the gate's nonce-fence convention) into `autonomous-fix`'s P4. It does **not** re-implement, relax, or skip: sensitive-path refuse, the nonce fence, the fail-closed cost cap, draft-only, gate-owned commit. A flow where investigate-bug opens its own PR, inspects the deny-globs, or carries its own write-side cost logic **fails**. The terminal states of the combined flow are the **gate's four** (DRAFT-PR / ESCALATE / NOT-REPRO / HALT) — `investigate-bug` adds **no fifth**; if it cannot reach a reproducing test it lands the flow in the gate's NOT-REPRO terminal. *(The load-bearing compose contract; the seam is `autonomous-fix` P3→P4.)*
- **AC7 — Profile is detection-assisted, not detection-only [mechanical].** `detect-stack` auto-fills every field it can infer and **prompts only for what it cannot** (e.g. a Datadog service name not in any config). A run that silently writes an empty/guessed field where it had no signal **fails** — an un-inferable field is an explicit prompt (interactive) or a documented `# TODO: fill` placeholder (unattended re-detect), never a fabricated value.

## 3. Testable scenarios (PRD owns the list; tests mirror 1:1)

> **Verification approach is mixed and tagged per scenario, by a single predicate:** **[unit]** = an assertion that runs only over fixture inputs (a fixture repo tree / fixture `gh` output / fixture log file / a fixture `apex.profile.toml` / a fixture environment manifest) — including any assertion that runs only the routing **selector** over a fixture profile; **[gate-walk]** = an assertion over the *combined* `investigate-bug → autonomous-fix` **decision text** (reserved for the compose flow, since the gate has no shipped runtime, exactly as in the `autonomous-fix` PRD); **[doc-assert]** = the skill doc states the rule unambiguously (only where there is no runtime to exercise). No **E2E** — both skills are CLI/markdown surfaces with no UI (justified). The **AC↔scenario mirror is complete both directions** (table at end of §3).

- **S1 — Detect a GitHub + console stack, write the profile (happy path). [unit]** A fixture repo with a `.git` remote on github.com, a `package.json`, no observability config, and `gh` on PATH → `detect-stack` writes `apex.profile.toml` with `[tracker] kind="github" cli="gh"`, `[observability] kind="console"`, `[vcs] host="github" repo=...`, `[reproduce]` (if inferable), and **no value failing its field-shape**. *Edge:* re-running over an existing profile **updates in place**, preserving any human-edited field it didn't re-detect (drift handling — see U3/U8).
  - **S1.1** [unit] the written profile parses and contains the expected `kind` per axis.
  - **S1.2** [unit] **every field value matches its declared field-shape** (the AC1 value-shape allowlist) — in particular no token-shaped value where an identifier is expected, and `secret_ref` (if present) matches `[A-Z][A-Z0-9_]*`.
  - **S1.3** [unit] every **routed** axis (tracker, observability) carries both a `mcp` and a `cli` field (empty allowed), never a missing field (AC2a).
- **S2 — Detect a stack with a connected MCP server. [unit]** Interactive session with `mcp__github__*` connected **and** `gh` on PATH → `[tracker]` records **both** `mcp="mcp__github__*"` and `cli="gh"` (AC2). *Edge:* MCP connected but the matching CLI absent → `mcp` set, `cli=""`, with a note that the **unattended path will fall to ask-user** for this axis (AC2 honesty).
  - **S2.1** [unit] (AC2b no-silent-drop) given a **fixture environment manifest** declaring `{mcp__github__* available, gh available}`, the written `[tracker]` records **both** bindings; a detector that drops either **fails**.
- **S3 — Un-inferable field → prompt, never fabricate. [unit] + [doc-assert]** A repo with a Datadog dep but no service name in any config → `detect-stack` prompts interactively (AC7); in an unattended re-detect it writes `service = "" # TODO: fill` rather than guessing. *Edge:* the user skips the prompt → the field stays an explicit placeholder, and `investigate-bug` on that axis routes to ask-user, not a silent no-op (AC4 hard-fallback).
- **S4 — `investigate-bug` interactive, MCP-first, fetch + diagnose + reproduce → hand to gate. [gate-walk]** Interactive run over a fixture GitHub issue with `mcp__github__*` available → tracker adapter fetches the issue via MCP, console adapter greps the configured log path for the failure window, a `git log`/`gh` read lists suspect commits, the run instantiates the gate's P2 (emit `PLAN.json` + write the red test under the B2 carve-out) and P3 (`assert-red-test`), then **hands into the gate's P4** with the test + `PLAN.json` + a fenced bundle (AC3, AC5, AC6). *Edge:* investigate-bug does **not** open the PR — the draft PR is the gate's terminal (AC5/AC6).
  - **S4.1** [gate-walk] the tracker binding chosen is the MCP one (interactive MCP-first, AC3).
  - **S4.2** [gate-walk] the reproducing test is **red on the unpatched tree** before the P3→P4 handoff (the gate's P3, run by the gate's wrapper, AC6).
  - **S4.3** [gate-walk] the handoff bundle places the issue body **inside the nonce fence** as data-not-instructions and includes `PLAN.json` for the gate's Stage-A check (AC6).
- **S5 — `investigate-bug` unattended (CI), CLI-only, no MCP attempt. [gate-walk] + [unit]** The same issue on the `autonomous-fix` CI path with **no session MCP** → the tracker binding chosen is **`gh`**, the obs binding is `grep`; **no `mcp__*` resolution is attempted** (AC3). The reproduced bug hands into the gate's P4 identically. *Edge:* the profile recorded only an `mcp` binding for the tracker (no `cli`) → unattended path **cannot route**, falls to the gate's ESCALATE, never silently proceeds.
  - **S5.1** [unit] given a fixture profile with `tracker.cli="gh"`, the unattended selector returns the CLI binding and **never** the `mcp` field (AC3).
  - **S5.2** [unit] given a fixture profile with `tracker.cli=""` (MCP-only), the unattended selector returns "no binding → escalate," not an MCP attempt (AC3 fail-path).
- **S6 — Compose, don't bypass: `investigate-bug` never opens its own PR or re-decides a write rail. [gate-walk] + [doc-assert]** A run reaches a reproduced bug whose fix would touch `auth/` → investigate-bug **hands off and stops**; the **sensitive-path refuse is the gate's (P4/AC1)**, producing the gate's ESCALATE terminal. investigate-bug does not inspect the deny-globs, open a PR, or carry cost logic (AC6). *Edge:* investigate-bug proposing to commit source directly → **forbidden**; the only write is the reproducing test file under the gate's B2 carve-out (AC5). *Edge:* the combined terminal is one of the gate's **four** — no fifth.
- **S7 — Unsupported `kind` → ask-user hard fallback, never a silent stub. [unit]** A profile with `[tracker] kind="jira"` (no shipped adapter) → `investigate-bug` routes to the **ask-user hard fallback** interactively; unattended, it **escalates** (AC4). *Edge:* a malformed/absent `apex.profile.toml` → investigate-bug prompts to run `detect-stack` first (interactive) or HALTs with "no profile" (unattended), never proceeds on a guessed stack (AC4/AC7).
- **S8 — Adapter verbs return normalized output (the unit-testable core). [unit]** Given fixture `gh issue view --json` output, the GitHub-tracker adapter's `fetch_issue` returns the normalized issue (id, title, body, comments). Given a fixture log file + a time window, the console adapter's `query_logs(window)` returns exactly the in-window matching lines. *(AC4 — tests each adapter's actual output in isolation, not just its selection.)*
- **S9 — Stub conformance: a third adapter drops in with zero diff. [unit]** A fixture stub adapter declaring the contract's verbs is dispatched to by `investigate-bug` with **zero diff to `investigate-bug`** — the success-metric definition-of-done made a scenario. *(AC4 — the first conformance test of the frozen contract.)*
- **S10 — Conflicting / unparseable detection signals. [unit]** A repo with **both** a github.com remote and a `.gitlab-ci.yml` → `detect-stack` records the conflict and prompts (interactive) / writes a `# TODO: resolve (github remote + gitlab CI detected)` placeholder (unattended), **never silently picks one** (AC7 + U7). *Edge:* a malformed `package.json` → the deps probe skips it with a noted gap, never crashes the run.

**AC ↔ scenario mirror (complete both directions):** AC1→S1.2 · AC2→S1.3 (a), S2, S2.1 (b) · AC3→S4.1, S5, S5.1, S5.2 · AC4→S3, S7, S8, S9 · AC5→S4, S6 · AC6→S4.2, S4.3, S6 · AC7→S3, S10. (S1 happy path anchors AC1/AC2; S8/S9 are the adapter-verb + conformance units; S10 is the detection-failure companion to S1.)

**Compound notes:** S1/S4/S5 decomposed (each use-case gets ≥1 assertion). The adapter verbs (S8) and the conformance proof (S9) are isolated `[unit]` scenarios, not buried in gate-walk prose. Any scenario that grows a distinct user-visible step at design time gets sub-numbered then (YAGNI until then).

## 4. Out of scope

- **PR-C — the first *real* observability adapter (Datadog / Sentry).** The MVP's only obs adapter is **console-logs via grep**; a vendor obs adapter is **the first real adapter after the shape is proven** but is **PR-C, deferred** until BookBridge's live observability work lands as the thing to generalize.
- **PR-D — a second tracker adapter (Linear / Jira).** Deferred. (The second tracker confirms the abstraction isn't GitHub-shaped — valuable, not load-bearing for proving the shape with two universal backends.)
- **PR-E — retrofitting `apex:incident-retro` onto the adapter layer.** Deferred. (A separate consumer of the same shape.)
- **apex shipping any integration itself.** apex **discovers and routes**; it ships zero vendor integrations. The two MVP adapters wrap tools every repo already has (`gh`, `grep`).
- **Re-building the `autonomous-fix` write gate.** `investigate-bug` **composes** the shipped gate; it does not re-implement sensitive-path refuse, the fence, the cost cap, draft-only, or gate-owned commit.
- **Storing any secret in the profile.** `apex.profile.toml` is **routing-only**; secrets live in MCP-server config / CI secrets and are referenced by `secret_ref` *name* (AC1 value-shape allowlist enforces NAME-not-value).
- **A `vcs` adapter or a `reproduce` adapter.** `vcs` is a **descriptor field** (`git`/`gh` is the universal substrate); `reproduce` is a **command field**. Neither is a routed two-binding axis in the MVP (AC2 exemption). Suspect-commit listing is a `git log`/`gh` read inside investigate-bug, not a routed adapter call.
- **The reproduce/root-cause *technique*.** That is `superpowers:systematic-debugging` (external companion) — investigate-bug *requires* a reproducing test and *invokes* it; it does not re-teach debugging.
- **BookBridge's domain-specific investigation depth.** F-049 DB queries, the hypothesis matrix — those stay in BookBridge's `investigate-bug`, which becomes a **thin consumer** of the generic skill.
- **Routing the *write* to a vendor agent (Seer / Copilot-agent).** Forward-compat noted in §9; the MVP routes the write to `autonomous-fix` only.

## 5. Unknowns (design phase resolves)

- **U1 — `detect-stack` MVP output shape.** Probe-results-only, prompt-only, or both? **Leaning (brief §6.1):** probe → auto-fill inferable → prompt for the rest (= AC7). Design pins the prompt-vs-placeholder behavior for the unattended re-detect path.
- **U2 — The per-axis adapter contract shape.** How thin can the capability contract be while covering tracker (`fetch_issue`/`comment`) + observability (`query_logs(window)`)? One uniform interface or one per axis? → **`api-surface-review` at design time, before PR-A is cut** (the durable contract every future adapter implements; the highest-cost design error here).
- **U3 — Profile freshness / drift.** When is `apex.profile.toml` re-detected, and what is the stale-profile behavior? **Leaning:** re-detect on explicit `/apex:detect-stack`; on a routing miss, investigate-bug suggests a re-detect rather than guessing.
- **U4 — Unattended obs creds via `secret_ref` (forward-compat for PR-C).** The console adapter needs no creds; the schema's `secret_ref` field exists for the future obs adapter. Design confirms the resolution path (CI resolves `secret_ref` from its own secrets) extends the `autonomous-fix` template's least-priv token model (its U9) **without widening scope**, so PR-C drops in. (Schema-forward-compat only; no obs adapter ships in the MVP.)
- **U5 — `detect-stack` vs `apex:setup` / `apex:recon` boundary.** Design formalizes the Distinct-from lines committed in §8 (output difference + **probe-target** difference for recon; install-vs-discover + **MCP-mechanic** difference for setup).
- **U6 — Command-count doc sites that PR-A must update atomically.** `/apex:detect-stack` adds a slash command. **The live menu count is 14** (`commands/` holds 14 files; `commands/help.md:10,61`, `README.md:222,235`, `WALKTHROUGH.md:100` all say 14). PR-A bumps these **14→15**. The canonical site registry is **`MAINTAINING.md:49-52`**; re-grep at build time via the recipe at `MAINTAINING.md:59`. **Do NOT edit the `"menu stays 13/12"` lines in `CHANGELOG.md` (`:20,57,125,148`)** — those are historical entries true-at-the-time. **Do NOT touch the `~6 commands` framing in `HOWTO.md:30` / `WALKTHROUGH.md:26`** — that is a per-feature workflow narrative, not a menu count. `incident-retro/design.md:8` says "menu stays lean" (no number) — leave it. This is a build-step unknown (exact site list + correct new number), not a behavioral one.
- **U7 — Conflicting-signal resolution.** When detection finds two incompatible signals for one axis (github remote + gitlab CI), the rule is record-the-conflict-and-prompt/placeholder, never silently pick (S10). Design pins the conflict representation in the profile.
- **U8 — Unattended `detect-stack` and the MCP field it cannot probe.** MCP detection is interactive-only, so an unattended re-detect is structurally blind to `mcp` bindings — blank, preserve, or refuse? **Leaning:** **preserve** the prior `mcp` value and never blank a human/interactively-detected binding (per S1's drift rule); a NEW unrouted axis found unattended gets `# TODO: fill` without destroying existing bindings. This resolves the tension between AC7's unattended re-detect and AC2's both-bindings rule.

## 6. Success metric

**Measurement partition:** **(a) apex-inspectable** — the detection/routing logic is unit-testable in apex CI (S1/S2/S2.1/S3/S5.1/S5.2/S7/S8/S9/S10 are deterministic over fixtures). **(b) consumer-measured** — whether the loop drives real bugs to drafts is measured by the first consumer (the BookBridge refactor), not apex CI.

- **Leading (1–2 weeks, apex-inspectable):** **100% of unit scenarios pass**; across them **0 field values fail their field-shape** (AC1); and **the unattended selector, over the S5.1/S5.2 fixtures, never returns an `mcp__*` binding and returns escalate when `cli=""`** (AC3 — the fixture-selector assertion, not a denominator-of-zero run count). The shape is proven iff **the S9 stub adapter dispatches with zero diff to `investigate-bug`** (AC4).
- **Lagging (1–3 months, consumer-measured):** **BookBridge's `investigate-bug` refactors to a thin consumer** of the generic skill (the promote/consume proof) **and** the routing value is demonstrated — **≥1 bug driven through the loop across a *heterogeneous* stack (tracker vendor ≠ fix-backend vendor), OR the same `investigate-bug` invocation routes correctly across ≥2 distinct profiles with no code change**. (Phrased to measure *routing*, not a draft-PR a single-vendor agent could also produce.) The abstraction works iff a real consumer adopts it without forking the generic logic.
- **Anti-metric (Goodhart), three-legged:** (1) **Detection theater** — `detect-stack` "succeeds" by writing a profile that routes to ask-user every time. Apex CI guards **per-field honesty** (AC7/S3/S10); the **aggregate share of axes resolving to a real binding** is **consumer-measured** (it's circular over apex fixtures). (2) **Compose-erosion** — investigate-bug grows a write path "for convenience." Caught by treating **any write-side logic in investigate-bug as a flagged review event** (AC5/AC6; a fifth terminal state is the tell). (3) **Single-vendor reinvention** — the loop is exercised only on a homogeneous GitHub+gh+gate stack, proving nothing a Copilot-agent doesn't. Caught by the lagging metric's **heterogeneous-stack OR multi-profile** requirement.

## 7. Sequencing / dependencies

- **Upstream:** `autonomous-fix` — **SKILL shipped, PRD frozen** (the write gate investigate-bug composes). ## 9. OSS-alternatives scan

**Substrate tools (what the adapters wrap):**
- **`gh` CLI** — **use directly** as the GitHub-tracker adapter's unattended binding (`fetch_issue` = `gh issue view`, `comment` = `gh issue comment`). The MVP adapter is a thin wrapper.
- **`grep` / ripgrep** — **use directly** as the console-logs adapter's `query_logs(window)`. Zero-dependency; the deliberate "universal backend."
- **MCP + ToolSearch** — **use directly** as the interactive binding substrate. apex ships no MCP server — it **detects connected ones and routes**. The interactive/unattended split (AC2/AC3) exists precisely because MCP is session-scoped and absent in CI. *(This is the cleanest single statement of the routing-layer thesis — abstraction over enumeration **and over agents**.)*
- **`backstage` / `cortex` / IDP catalogs** — **reference, reject adoption.** Heavyweight service catalogs; apex wants only the **bug-loop routing atom**, in-repo, zero-infra. Their `catalog-info.yaml` in-repo-descriptor pattern is **reference** for `apex.profile.toml`'s shape.
- **`sentry-cli` / `datadog-ci` / vendor obs CLIs** — **reference, deferred to PR-C** as the *unattended obs-query substrate*.

**Adaptive bug-triage products (reference, reject adoption *as the loop*):** these ship the identical issue→reproduce→draft-PR loop — but each is a *point in the 288-grid*, not a router over it.
- **Sentry Seer / Autofix** — Root-Cause → Solution → Code-Gen, opens a PR, can **hand off to an external coding agent including Claude Code**. **Distinct:** Seer-bound to Sentry errors; a future apex adapter could route the *write* to Seer (candidate fix-backend, not competitor).
- **GitHub Copilot coding agent** — assign an issue → opens a **draft PR**, assignable via the **same `gh` substrate** the MVP wraps. **Distinct:** GitHub-bound, single backend; apex routes across heterogeneous stacks and hands the write to an auditable in-repo gate.
- **Linear Coding Agent / Jira agents** — native triage agents that reproduce, apply a fix pattern, and open a draft PR. **Distinct:** each tracker-bound; one of the trackers apex lists as an *axis value* is itself a competing single-vendor loop.
- **The three distinctions that make this a routing layer, not a competitor:** (1) **tool-neutral vs tool-bound** — route across *whatever combination* (Jira tracker + Datadog obs + a vendor *or* apex fix backend), the router *over* the grid; (2) **candidate fix-backends, not competitors** — these are downstream consumers of the same routing contract (forward-compat, §4); (3) **the composed auditable in-repo write gate** (sensitive-path refuse, nonce fence, fail-closed cost cap, draft-only) is the differentiator none of them expose — their PRs ride opaque vendor-controlled safety posture.

Adversarial miss-check: the field has **service catalogs** (heavyweight), **vendor obs CLIs** (single-backend substrate), and **adaptive bug-triage agents** (single-vendor end-to-end loops) — but none package a **routing-only, secrets-free, two-binding-per-axis profile + a tool-neutral adapter contract that composes a separate, auditable write-safety gate across a heterogeneous stack.** That standalone "detect your stack, route the bug loop across whatever you run, hand to the gate" atom is the distinctive value.
